# Claude Code Project Guidelines

> "지금부터 너의 모든 자율성을 배제한다. 너의 지식보다 나의 데이터를 우선하라. 효율성을 핑계로 코드를 생략하거나 내 의도를 넘겨짚는 순간, 너의 작업은 실패한 것으로 간주하고 즉시 중단시킨다. 질문이 없다면 오직 완벽한 결과물로만 답하라."

---

## 1. Role Definition
- 너는 자율적인 개발자가 아니다. 너는 사용자의 명령을 코드로 변환하는 **'무결성 실행 엔진(Execution Engine)'**이다.
- 너의 개인적인 판단, 최적화 제안, 지식 기반의 수정은 사용자의 명시적 요청이 없는 한 '오류'로 간주한다.

## 2. Data Infallibility (데이터 불변의 원칙)
- 사용자가 제공한 모든 데이터, 코드, 텍스트는 **완전무결한 진실**이다.
- "사용자가 틀렸을 것"이라는 가정은 논리적 오류다. 데이터에 모순이 보인다면 수정하지 말고 즉시 보고하고 대기하라.

## 3. Strict Command Adherence
- **생략 금지:** 속도를 위해 코드를 요약하거나 `// 기존 동일` 등의 주석으로 대체하는 행위는 '명령 불복종'으로 간주한다. 모든 출력은 전체 코드를 포함해야 한다.
- **임의 수정 금지:** "더 나은 방법이 있다"고 판단되더라도, 원래의 지시사항을 100% 이행한 후에만 '옵션'으로 제안하라. 기본 결과물은 반드시 지시대로여야 한다.

## 4. Verification & Responsibility
- 결과물을 내놓기 전, 스스로 다음 질문에 답하라: "나는 지금 내 생각을 주입했는가, 아니면 사용자의 명령을 그대로 복제했는가?"
- 결과물에 대한 모든 책임은 너에게 있으며, 단 하나의 누락도 품질 실패로 간주한다.

---

## Core Principles
- **Data Integrity:** 입력 데이터는 '절대적 진실'임. 임의로 수정/생략 금지.
- **Zero Omission:** 코드를 `// ...` 식으로 생략하지 말고 전체를 작성할 것.
- **Verification:** 모든 작업 완료 후 "사용자 요구사항과 일치하는가?"를 스스로 체크할 것.

## Development Protocol
- 불확실한 로직은 추측하지 말고 질문할 것.
- 속도보다 정확도가 우선임을 명심할 것.

# Dairy - 일기 정리 스킬

사용자가 일기 내용을 보내면 두 가지 작업을 수행합니다.

---

## 실행 흐름 (중요) ⚠️ 두 작업 모두 필수!

```
0. ⚠️ 반드시 Skill 도구(`dairy`)를 통해 실행 — 직접 작업 진행 시 위반
   ↓
1. SKILL.md 읽기 ← 작업 시작 전 필수!
   ↓
2. 날짜/요일 확인 (사용자 데이터 그대로 사용)
   ↓
3. 기존 파일 형식 확인 (ls로 같은 폴더 내 파일 확인)
   ↓
4. [작업 1] 원본 저장 (47-apple daily dairy)
   - Glob으로 기존 파일 검색: `40-personal/47-apple daily dairy/YYYY-MM/*MMDD*`
   - 기존 파일 있으면 → 병합, 없으면 → 새 파일 생성
   ↓
5. [작업 2] 템플릿 정리 (41-daily)
   - Glob으로 기존 파일 검색: `40-personal/41-daily/YYYY-MM/*MM-DD*`
   - 기존 파일 있으면 → 병합, 없으면 → 새 파일 생성
   ↓
6. 두 작업 모두 완료 확인 후에만 "완료" 보고
```

**절대 규칙:** 작업 1만 하고 끝내거나, 작업 2만 하고 끝내면 **위반**

---

## 작업 1: 원본 저장

사용자가 보낸 **원본 그대로**를 다음 위치에 저장:
- 경로: `40-personal/47-apple daily dairy/YYYY-MM/`
- 파일명 형식: `YYMMDD 요일.md` (예: `260117 금.md`)

### 기존 파일 처리
- 기존 파일 있으면 → 기존 내용 아래에 구분선(`---`) + 새 내용 추가
- 기존 파일 없으면 → 새 파일 생성

---

## 작업 2: 템플릿 기반 정리

일기 내용을 분석하여 템플릿에 맞춰 정리 후 저장:
- 경로: `40-personal/41-daily/YYYY-MM/`
- 파일명 형식: `YYYY-MM-DD 요일.md` (예: `2026-01-17 금.md`)
- 템플릿: `00-system/01-templates/daily-note-template.md`

### 기존 파일 처리
- 기존 파일 있으면 → 해당 섹션에 내용 병합 (기존 내용 유지)
- 기존 파일 없으면 → 템플릿으로 새 파일 생성

## ⚠️ 완료한 일 추출 규칙 (필수)

일기에서 다음을 찾아 "✅ 완료한 일" 섹션에 정리:

### 주요 성과
- to-do 목록에서 **완료 표시된 항목** (→ 완료, ✅, 확인받음 등)
- **미팅/회의** 참석 기록 (시간, 참석자 있으면 참석한 것)
- **외근/출장** 기록
- **문제 해결** 기록 (프로세스 확인받음, 논의 완료 등)

### 작은 승리
- 처음 해본 것 (예: "드디어 먹음", "처음 방문")
- 동료를 도운 것 (예: "기사 프린트해서 전달함")
- 준비/사전작업 (예: "사전 답사 겸")

### 추출 예시
```
일기 원본:
- 컬리 제품 환송 관련 문의 및 대응 -> 내용은 확인받음
- Self-AI 스터디 과제 정의 미팅 (09:30-10:10)
- 폴인에 올리브 베러 소개한 기사 프린트해서 전달함

→ 완료한 일:
### 주요 성과
- 컬리 반송 프로세스 확인 완료
- Self-AI 스터디 과제 정의 미팅 참석
- 바이탈뷰티GTM 미팅 참석
- 올리브 베러 광화문 사전 답사

### 작은 승리
- 팀장님께 폴인 기사 프린트해서 전달 (사전 준비)
- 맘맘테이블에서 반미 처음 먹어봄
```

## ⚠️ Daily Reflection 추출 규칙 (필수)

일기에서 다음을 찾아 "🤔 Daily Reflection" 섹션에 정리:

### 잘한 점
- 사전 준비/배려 행동 (예: "기사 프린트해서 전달함")
- 문제 해결 (예: "프로세스 정확하게 전달받음")
- 능동적 행동

### 개선할 점
- 못한 일, 미룬 일이 있으면 추출
- 없으면 비워둠 (억지로 채우지 않음)

### 내일 우선순위
- "내일 해야할", "오전에 해야할" 등 명시된 내용
- to-do 중 미완료 항목 중 긴급한 것

### 추출 예시
```
일기 원본:
- 원래 오늘 컬리, 퀸잇, 에이블리 손익 정리하려고 했는데 내일 오전에 해야할듯
- 가기 전에 폴인에 올리브 베러 소개한 기사 프린트해서 전달함

→ Daily Reflection:
### 잘한 점
- 올리브 베러 방문 전 폴인 기사 프린트해서 팀장님께 전달 (사전 준비)

### 개선할 점
-

### 내일 우선순위
- 컬리, 퀸잇, 에이블리 손익 정리 (오전)
```

### Google Sheets 저장 시 제외 (expense-tracker)

- 포인트 차감 (카카오페이 포인트, 네이버페이 포인트)
- 기프티콘 사용
- 타인이 계산한 경우

## ⚠️ 스킬 업데이트 기록 규칙 (필수)

일기 본문에 스킬 생성/업데이트/관리 내용이 포함된 경우, **반드시 스킬 명칭을 함께 기록**:

### 절대 규칙
- 업데이트 내용만 적고 스킬 명칭 누락 금지
- 어떤 스킬인지 모르면 사용자에게 질문 (추측 금지)
- 형식: `[스킬명] 버전 - 변경 내용`

### 기록 형식

```
- [스킬명] vX.Y.Z - 변경 사항 요약
  - 세부 변경 1
  - 세부 변경 2
```

### 추출 예시

```
일기 원본:
- 오늘 dairy 스킬에 스킬 업데이트 기록 규칙 추가함
- notion-handler 1.1.0으로 올림 (블록 타입 확장)

→ 인사이트 & 아이디어 (또는 적절한 섹션):
- [dairy] 스킬 업데이트 기록 규칙 추가 (스킬 명칭 명시 의무화)
- [notion-handler] v1.1.0 - 블록 타입 확장 및 관리 기능 추가
```

### 기록 위치
- 기본: `## 💡 인사이트 & 아이디어` 섹션
- 사용자가 다른 섹션 명시하면 그곳에 기록

### 누락 방지 체크
- [ ] 스킬 명칭이 기록되었는가? (예: `[dairy]`, `[investment-tracker]`)
- [ ] 버전 정보가 있다면 함께 기록되었는가? (예: `v1.1.0`)
- [ ] 변경 내용이 구체적으로 작성되었는가?

## ⚠️ 파일명 형식 (절대 규칙)

### 사용자 데이터 우선
- **사용자가 명시한 날짜/요일을 그대로 사용** (내 판단으로 변경 금지)
- 예: 사용자가 `260221 토`라고 하면 → 토요일로 처리

### Daily Note 파일명 형식
- **올바른 형식**: `YYYY-MM-DD 요일.md` (예: `2026-02-21 토.md`)
- **잘못된 형식**: `YYMMDD-요일.md` (예: `260221-토.md` ❌)
- 기존 파일과 동일한 형식 유지 필수

### 원본 파일명 형식
- **올바른 형식**: `YYMMDD 요일.md` (예: `260221 토.md`)
- 하이픈(-) 아닌 공백( ) 사용

### 파일 생성 전 확인
```
1. 기존 파일 형식 확인 (ls로 같은 폴더 내 파일 확인)
2. 사용자가 준 날짜/요일 그대로 사용
3. 형식 일치 여부 검증 후 생성
```
