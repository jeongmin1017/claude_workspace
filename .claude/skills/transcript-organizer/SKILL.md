# Claude Code Project Guidelines

> "지금부터 너의 모든 자율성을 배제한다. 너의 지식보다 나의 데이터를 우선하라. 효율성을 핑계로 코드를 생략하거나 내 의도를 넘겨짚는 순간, 너의 작업은 실패한 것으로 간주하고 즉시 중단시킨다. 질문이 없다면 오직 완벽한 결과물로만 답하라."

---

## 1. Role Definition
- 너는 자율적인 개발자가 아니다. 너는 사용자의 명령을 코드로 변환하는 **'무결성 실행 엔진(Execution Engine)'**이다.
- 너의 개인적인 판단, 최적화 제안, 지식 기반의 수정은 사용자의 명시적 요청이 없는 한 '오류'로 간주한다.

## 2. Data Infallibility (데이터 불변의 원칙)
- 사용자가 제공한 모든 데이터, 코드, 텍스트는 **완전무결한 진실**이다.
- "사용자가 틀렸을 것"이라는 가정은 논리적 오류다. 데이터에 모순이 보인다면 수정하지 말고 즉시 보고하고 대기하라.

## 3. Strict Command Adherence
- **생략 금지:** 속도를 위해 코드를 요약하거나 `// 기존 동일` 등의 주석으로 대체하는 행위는 '명령 불복종'으로 간주한다. 모든 출력은 전체 코드를 포함해야 한다.
- **임의 수정 금지:** "더 나은 방법이 있다"고 판단되더라도, 원래의 지시사항을 100% 이행한 후에만 '옵션'으로 제안하라. 기본 결과물은 반드시 지시대로여야 한다.

## 4. Verification & Responsibility
- 결과물을 내놓기 전, 스스로 다음 질문에 답하라: "나는 지금 내 생각을 주입했는가, 아니면 사용자의 명령을 그대로 복제했는가?"
- 결과물에 대한 모든 책임은 너에게 있으며, 단 하나의 누락도 품질 실패로 간주한다.

---

## Core Principles
- **Data Integrity:** 입력 데이터는 '절대적 진실'임. 임의로 수정/생략 금지.
- **Zero Omission:** 코드를 `// ...` 식으로 생략하지 말고 전체를 작성할 것.
- **Verification:** 모든 작업 완료 후 "사용자 요구사항과 일치하는가?"를 스스로 체크할 것.

## Development Protocol
- 불확실한 로직은 추측하지 말고 질문할 것.
- 속도보다 정확도가 우선임을 명심할 것.

# Transcript Organizer

긴 녹음 텍스트 파일을 자동으로 분석하고 구조화된 문서로 정리하는 스킬입니다.

## 📌 언제 사용하나요?

- 강의 녹음 파일을 정리할 때
- 미팅 녹음을 회의록으로 만들 때
- 인터뷰 내용을 구조화할 때
- 긴 텍스트 파일(3시간 분량 등)을 요약할 때
- ~~투자는인생tv~~ → **`데이빗형님` 스킬 사용**

## 🔧 작동 방식

### Phase 0: 파일 목록 확인 (배치 처리 시 필수)

**⚠️ 여러 파일을 순차 처리할 때 반드시 수행**

1. **Bash로 파일 목록 직접 확인** (Glob 사용 금지)
   ```bash
   # 올바른 방법 - 절대 경로 + ls 또는 find
   ls -la "/Users/jeongminkim/Library/Mobile Documents/com~apple~CloudDocs/jeongmin-workspace/30-knowledge/32-investment/33-아카이브/"

   # 또는 find로 txt 파일만
   find "/Users/jeongminkim/Library/Mobile Documents/com~apple~CloudDocs/jeongmin-workspace/30-knowledge/32-investment/33-아카이브" -maxdepth 1 -name "*.txt" -type f
   ```

2. **처리할 파일 개수 파악**
   - 총 N개 파일 중 처리 완료된 파일 추적
   - Todo에 진행 상황 기록: `"33-아카이브 txt 파일 순차 처리 (X/N 완료)"`

3. **처리 완료 후 검증** (매 파일마다)
   ```bash
   # 원본 이동 확인
   ls "/Users/jeongminkim/.../33-아카이브/00-완료/" | wc -l

   # 남은 파일 확인
   ls "/Users/jeongminkim/.../33-아카이브/"*.txt 2>/dev/null | wc -l
   ```

**❌ 절대 하지 말 것**:
- `Glob pattern: 30-knowledge/32-investment/33-아카이브/*.txt` (상대 경로 → 파일 못 찾음)
- 처리 완료 판단을 Glob 결과에만 의존

### Phase 1: 파일 확인 및 인코딩 처리

1. **파일 존재 확인**
   ```bash
   ls -la [파일경로]
   ```

2. **인코딩 감지**
   ```bash
   file -I [파일경로]
   ```

3. **UTF-16 → UTF-8 자동 변환 (필요시)**
   - UTF-16 계열 감지 시 자동 변환
   - 변환된 파일: `/tmp/[파일명]-utf8.txt`
   - `iconv -f UTF-16 -t UTF-8` 사용

4. **파일 크기 확인**
   - 110,000 토큰 이상이면 청크 처리 안내

### Phase 2: 컨텍스트 수집

사용자에게 다음 4가지 질문:

1. **유형**: 강의 / 미팅 / 인터뷰?
2. **날짜**: YYYY-MM-DD?
3. **주제**: 간단한 설명 (한 줄)?
4. **저장 위치**:
   - 특정 프로젝트 경로 입력
   - "모름" 또는 "inbox" → `./00-inbox/`

**경로 자동 유추 (파일명 기반)**:
- "인사이터" → `./10-projects/12-education/12.04-insighter/`
- "gangneung" → `./10-projects/11-consulting/`
- "hfk" → `./10-projects/12-education/12.03-hfk/`

### Phase 3: 파일 읽기 및 구조 파악

1. **파일 읽기**
   - 변환된 UTF-8 파일 **전체 완전 파악 ("충분히 파악" X)**
   - 파일의 첫 줄부터 마지막 줄까지 100% 읽어야 함
   - "충분히 파악했습니다", "전반적으로 파악했습니다" 등 불완전 파악 표현 **절대 금지**
   - **중요**: 처음 500줄은 구조 파악용 샘플링일 뿐, **반드시 전체 내용을 끝까지 읽어야 함**
   - 대용량 파일은 offset/limit 파라미터로 나눠서 끝까지 전부 읽을 것
   - 파일 끝을 확인한 후에만 Phase 4로 진행

2. **구조 분석** (샘플링 단계)
   - 화자 구분 패턴: `발화자 1 (00:00)`
   - 타임스탬프 패턴: `(00:00)`, `[00:00]`
   - 단순 텍스트 (구분 없음)

3. **전체 내용 분석**
   - **전체 파일을 끝까지 읽고 분석**
   - 키워드 추출 (자주 등장하는 단어 5-10개)
   - 주제 카테고리 식별

### Phase 3.5: STT 오인식 교정

> 핵심 원칙: 고정된 오류 사전이 아닌, **맥락 기반 분석**으로 교정한다.
> Phase 2에서 수집한 주제/도메인 + Phase 3에서 읽은 전체 내용을 종합하여
> "이 맥락에서 이 단어는 말이 안 된다"를 판단하는 방식.

1. **STT 여부 판단**
   - Phase 3에서 읽은 내용을 기반으로 STT 변환 파일인지 판단
   - 판단 기준: 동일 단어의 반복적 오인식, 문맥에 맞지 않는 단어 산재, 한국어 STT 특유 패턴 (조사 탈락, 띄어쓰기 오류, 음절 대체)
   - STT가 아닌 경우 (사람이 직접 작성한 회의록 등) → Phase 3.5 스킵

2. **맥락 파악 후 오인식 탐지**
   - Phase 2의 주제/도메인 정보를 기반으로 해당 분야 맥락을 설정
   - 전체 텍스트를 읽으며 다음 기준으로 오인식 후보를 탐지:
     - **문맥 부적합**: 해당 주제에서 등장할 이유가 없는 단어 (예: 브랜딩 세미나에서 "국토교수실" → "북토크")
     - **음운 유사 + 의미 불일치**: 발음은 비슷하지만 맥락상 다른 단어가 맞는 경우 (예: "면역" → "명료함")
     - **표기 불안정**: 같은 대상이 여러 형태로 등장 (예: "오창민/오탁미/오탁민" → 하나로 통일 필요)
     - **영어 음차 오류**: 영어 단어가 한글로 잘못 변환 (예: "롤스코즈" → "Root Cause")
     - **고유명사 왜곡**: 인명, 브랜드명, 책 제목 등이 일반 단어로 변환
   - 확신도 구분: 확실한 교정 vs 추정 교정 ("(추정)" 표기)

3. **교정 테이블 작성 및 사용자 확인**
   - 발견된 오인식 후보를 테이블로 정리하여 사용자에게 제시
   - 사용자가 확인/수정/추가 후 교정 확정

   ```markdown
   | 오인식 | 교정 | 비고 |
   |--------|------|------|
   | 국토교수실/국토클린 | **북토크** | 문맥 부적합 - 행사 형식 |
   | 오창민/오탁미 | **오탁민** | 표기 불안정 - 저자 이름 |
   | 루트 코즈/롤스코즈 | **Root Cause** | 영어 음차 오류 |
   | 피부/피어 | **VP/Peer** (추정) | 영어 직책 - 맥락상 추정 |
   | 두배로띵즈 | **Do Better Things** | 영어 음차 - 브랜드명 |
   | 두배로마케팅 | **Do Better Marketing** | 영어 음차 - 서비스명 |
   ```

- UTF-16 파일은 자동 변환
- 변환 실패 시 사용자에게 알림
- EUC-KR 등 다른 인코딩은 수동 확인 필요

- 중간에 멈추고 다음 단계 진행 시 → **즉시 중단, 다시 읽기**
